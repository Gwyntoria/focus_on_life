# 哪家 AI 最好用对我来说不重要

Tony 问我：Claude Code 和 Codex 用下来，你觉得哪个更好用？

这是一个很正常的问题。正常到几乎不好回答。

因为在公司语境里，工具当然要比较。谁更稳定，谁更便宜，谁更适合接入权限系统，谁更容易管理上下文，谁在代码仓库里更少出错，谁的审计链路更清楚，谁的响应速度更快，这些都不是小事。工具进入组织以后，就不再只是个人喜好，而是成本、效率、风险和责任的重新分配。

但如果把问题缩小到我自己，我的答案反而简单：哪家 AI 最好用，对我来说不重要。

这句话不是说工具没有差别。工具当然有差别。Claude Code 有它的长处，Codex 也有它的长处。一个人如果真的每天用，就会知道不同工具在上下文管理、代码修改、解释能力、执行节奏、错误修复上各有脾气。说它们完全一样，是懒；说某一家绝对胜出，是另一种懒。

问题在于，这个比较很容易把注意力引到错误的地方。

很多人讨论 AI 工具，像在讨论一把刀哪家更锋利。问题是，真正决定厨房能不能出菜的，常常不是刀锋，而是你知不知道自己要做什么菜，食材有没有洗干净，火候怎么控制，锅里冒烟时你是不是还在翻菜谱。

AI 也是这样。

对我来说，AI 首先不是一个「答案机器」，而是一个「追问机器」。我真正依赖它的地方，不是它第一次回答得多漂亮，而是它能不能被我反复追问、反复校正、反复拆解，直到我把问题看清楚。

第一次回答通常不重要。

第一次回答大概率只是一个入口。它会给出一个结构，列出几个可能方向，写一段看上去很完整的解释，或者直接生成一批代码。这个阶段最容易让人产生幻觉：我好像懂了，问题好像解决了，事情好像可以交付了。

现实通常不是这样。

现实里，第一次回答往往只是把模糊变成了整齐。整齐不是理解。整齐只是降低了你的焦虑，让你误以为混乱已经被消灭。

所以我的工作流不是「问一下，然后采纳」。我的工作流更接近审讯，或者更温和一点，叫复盘。

我会继续问：你为什么这么判断？这个结论依赖哪些前提？如果前提不成立会怎样？有没有反例？这里有没有偷换概念？这段代码放到现有项目里，会影响谁？异常路径在哪里？测试应该覆盖什么？你刚才的说法和前面那句是否矛盾？如果我要向同事解释，最短的版本是什么？如果我要自己以后维护，最危险的地方在哪里？

问到这里，工具差异仍然存在，但已经不是最重要的变量。

因为这个过程里，真正承担责任的人不是 AI。是我。

AI 可以给我答案，但它不能替我确认我到底懂没懂。它可以帮我写代码，但不能替我承担线上报错。它可以替我总结方案，但不能替我面对主管、同事、用户和后续维护成本。它可以把一个问题说得很顺，但顺不等于对，顺更不等于可交付。

这就是 AI 工具使用里最容易被忽略的一点：人不是在和工具比聪明，人是在通过工具暴露自己的理解边界。

如果我问不出第二个问题，说明我对第一个答案也没有真正理解。

如果我无法判断 AI 的回答哪里可疑，说明我不是在使用工具，而是在托管责任。

如果我只会比较「哪个模型更强」，却说不清自己在什么任务、什么约束、什么验收标准下需要它强，那这个比较就是消费评测，不是工作判断。

Tony 问 Claude Code 和 Codex 哪个更好用，这个问题在组织层面仍然有价值。公司需要选择工具，需要算账，需要管理权限，需要控制风险，需要建立统一流程。这是组织治理问题。

但对个人来说，更关键的问题应该换一个问法：你能不能把 AI 放进自己的工作流里，使它持续提高你的理解、判断和交付？

这个问题比「哪家最好用」难得多。

因为「哪家最好用」可以外包给评测文章、排行榜、朋友圈和同事口碑。今天这个强，明天那个强；今天这个上下文长，明天那个代理能力好；今天这个生成代码稳，明天那个重构更顺。工具市场像金融市场一样，短期波动会制造大量话题，但真正重要的是你的资产配置。

我的资产不是某个工具账号。我的资产是提问能力、验证习惯、领域理解和复盘记录。

这些东西不随某家公司模型更新而归零。

我用 Claude Code，也用 Codex。它们都帮我完成过需求，也都犯过错；它们都在我追问下把问题讲得更清楚，也都在某些时候给出过看似正确但实际需要校验的答案。到最后，我对它们的信任不是「相信它不会错」，而是「相信它能在我的追问和验证里，成为一个足够好的协作者」。

这两种信任不一样。

前一种信任是把方向盘交出去。后一种信任是你仍然握着方向盘，只是旁边多了一个能看地图、查资料、提醒路况、偶尔也会报错的副驾驶。

副驾驶很重要。但副驾驶不是司机。

所以我现在更愿意把 AI 当成一种工作中的反复对话机制。它的价值不止在生成，更在迫使我把含混的判断说清楚。很多时候，我以为自己只是让 AI 帮我写一段代码，最后发现自己其实是在补一块工程理解；我以为自己只是让它改一篇文章，最后发现自己是在重新确认文章的核心问题；我以为自己只是让它解释一个概念，最后发现我之前只是会使用那个词，并不真正知道它在什么边界下成立。

这才是对我有用的地方。

AI 把很多事情变快了，但速度不是唯一价值。速度有时甚至会遮蔽理解。过去不会的人做不出来，现在不会的人也能做出一个像样的东西。区别在于，像样之后，你能不能看出它哪里不像。

这要求人保留一点慢能力。

慢慢问，慢慢验，慢慢把问题拆到自己能承担的粒度。不是每个问题都要问到哲学深处，但至少要问到自己能解释、能修改、能负责。

如果只是写一段临时代码，标准可以低一点；如果要进生产环境，标准就要高一点。如果只是给自己看，容错可以大一点；如果要交给团队，边界就要写清楚。如果只是探索思路，可以允许发散；如果要做决策，就必须回到证据、代价和责任。

这不是工具偏好问题，而是工作伦理问题。

在 AI 时代，一个人很容易把「我能让 AI 做出来」误认为「我有能力做这件事」。这中间隔着一条沟。沟里是理解，是验证，是取舍，是承担后果。

能跨过去，AI 是杠杆。

跨不过去，AI 是麻醉剂。

所以，如果以后还要问我 Claude Code 和 Codex 哪个更好用，我大概会先反问：你要用它解决什么问题？这个问题的失败代价是什么？你需要的是更快生成，还是更好理解？你准备怎么验证它？你愿意把哪部分责任交给工具，哪部分责任必须留在自己手里？

问清这些以后，再谈工具，才有意义。

对我自己来说，答案暂时很朴素：两家都够用，两家都不够让我放弃追问。

够用，是因为它们都能在我不断追问时帮我完成需求。

不够，是因为任何 AI 都不能替我完成理解本身。

而工作里真正值钱的，常常不是第一个答案。

是你知道第一个答案还不够。
